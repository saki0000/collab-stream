# ADR 005: 段階的APIセキュリティ戦略

## Status
Accepted

## Context

CollabStreamではYouTube API、Twitch API との統合が必要で、APIキーの適切な管理が求められる：

**セキュリティ要件**:
- APIキー流出の防止
- 不正利用による課金回避
- サービス停止リスクの最小化

**制約条件**:
- 開発初期の低コスト運用
- 段階的なユーザー数増加想定
- サーバー運用の学習コスト

APIキー管理方式として以下を検討：

1. **環境変数方式**: ビルド時にAPIキーを注入
2. **プロキシサーバー方式**: サーバー経由でAPI呼び出し
3. **クライアント暗号化方式**: アプリ内でキーを暗号化保存
4. **ハイブリッド方式**: 段階的にセキュリティを強化

## Decision

**段階的APIセキュリティ戦略** を採用する。

### Phase 1: 環境変数 + レート制限 (開発初期)
- BuildConfig でAPIキーを管理
- アプリ内レート制限を実装
- 使用量監視を設定

### Phase 2: プロキシサーバー移行 (ユーザー増加時)
- 専用サーバーでAPIキー管理
- API呼び出しの一元制御
- 使用量制限とログ記録

### 採用理由

1. **段階的リスク管理**: 初期は低コスト、成長に応じて強化
2. **開発効率**: 複雑なサーバー構築を後回し
3. **移行可能性**: Phase 1→2 の設計で移行コスト最小化
4. **コスト効率**: ユーザー数に応じたインフラコスト
5. **学習コスト**: 段階的な技術習得

## Consequences

### Positive
- **開発速度**: 初期開発の高速化
- **コスト効率**: 段階的なインフラ投資
- **リスク分散**: 成長段階に応じたセキュリティ強化
- **技術学習**: サーバー運用の段階的習得
- **移行設計**: Phase間の移行が容易

### Negative
- **初期リスク**: Phase 1 でのAPIキー解析リスク
- **二重実装**: Phase 1→2 移行時の実装コスト
- **監視複雑性**: Phase 切り替え時期の判断

### Neutral
- **セキュリティレベル**: Phase 2 で十分な保護レベル到達
- **運用負荷**: Phase 2 移行後のサーバー運用コスト
